# NapCat 用 QQ 接入 OpenClaw 的安全模式教程

## 1. 教程目标

这份文档只讲技术实现，不包含人格、人设或调教内容。

目标是完成以下事情：

1. 使用 NapCat 将 QQ 消息接入 OpenClaw
2. 将 QQ 通道固定到受限代理，而不是主代理
3. 让 WebChat 保留完整管理权限
4. 让 QQ 侧只负责聊天，不具备本机操作权限
5. 为后续扩展图片识别、群聊规则、独立工作区预留结构

--------------------------------------------------

## 2. 关键路径

以下路径是这套方案里最关键的几个位置：

OpenClaw 主配置：
C:\Users\Administrator\.openclaw\openclaw.json

NapCat 插件目录：
C:\Users\Administrator\.openclaw\extensions\openclaw-napcat-plugin-master

WebChat 主工作区：
C:\Users\Administrator\.openclaw\workspace

QQ 安全模式工作区：
C:\Users\Administrator\.openclaw\workspace-qqsafe

QQ 工作区建议保留这些文件：
- C:\Users\Administrator\.openclaw\workspace-qqsafe\SOUL.md
- C:\Users\Administrator\.openclaw\workspace-qqsafe\AGENTS.md
- C:\Users\Administrator\.openclaw\workspace-qqsafe\USER.md
- C:\Users\Administrator\.openclaw\workspace-qqsafe\IDENTITY.md
- C:\Users\Administrator\.openclaw\workspace-qqsafe\memory\YYYY-MM-DD.md

--------------------------------------------------

## 3. 推荐架构

建议把整套链路拆成四层：

第一层：NapCat
负责接收 QQ 消息并暴露 webhook

第二层：OpenClaw 的 NapCat 插件
负责把 NapCat 的回调事件转成 OpenClaw 的输入

第三层：openclaw.json
负责指定：
- NapCat 走哪个代理
- 各代理有哪些工具权限
- gateway 如何暴露

第四层：代理对应工作区
负责决定：
- 各通道读哪个工作区
- 各工作区使用什么规则
- 各通道的边界与安全策略

如果只把消息打通，而不做权限隔离，最终很容易出现：
- QQ 能够诱导代理提供本机命令指导
- QQ 能暴露本机文件相关信息
- QQ 和 WebChat 共用同一套高权限能力

所以“能回消息”和“安全可用”不是一回事。

--------------------------------------------------

## 4. 推荐权限模型

推荐直接采用双代理模型：

main
给 WebChat 使用，保留完整能力

qqsafe
给 NapCat / QQ 使用，只负责聊天，禁止本机操作能力

最核心的原则是：

WebChat = 管理入口
QQ = 受限入口

不要让 QQ 直接跑 main。
真正稳的方式是技术隔离，而不是只靠提示词克制。

--------------------------------------------------

## 5. openclaw.json 推荐配置

下面给出推荐思路。
示例中省略了你的真实 API key、token 等敏感值，请自行填写。

### 5.1 代理配置

```json
{
  "agents": {
    "defaults": {
      "workspace": "C:\\Users\\Administrator\\.openclaw\\workspace"
    },
    "list": [
      {
        "id": "main"
      },
      {
        "id": "qqsafe",
        "workspace": "C:\\Users\\Administrator\\.openclaw\\workspace-qqsafe",
        "tools": {
          "deny": [
            "write",
            "edit",
            "group:runtime",
            "group:memory",
            "group:sessions",
            "group:automation",
            "group:nodes"
          ],
          "elevated": {
            "enabled": false
          }
        }
      }
    ]
  }
}
```

### 5.2 NapCat 通道固定走 qqsafe

```json
{
  "channels": {
    "napcat": {
      "url": "http://127.0.0.1:3000",
      "agentId": "qqsafe",
      "enableGroupMessages": true,
      "groupMentionOnly": true
    }
  }
}
```

如果你只想允许特定 QQ 用户触发，也可以额外加白名单：

```json
"allowUsers": ["2226062736"]
```

但如果你的目标是“所有 QQ 用户都能聊天、但没有电脑权限”，那么只要 qqsafe 的权限已经受限，就可以不启用 allowUsers。

### 5.3 gateway 建议

建议保持：
- bind = loopback
- auth = token

这样 WebChat 仍是管理入口，但不会裸暴露到公网。

--------------------------------------------------

## 6. qqsafe 代理到底禁掉了什么

这一套 deny 的目的，不是让它什么都不会，
而是让它只承担聊天职责，不碰本机。

建议禁用：

write
不能写文件

edit
不能改文件

group:runtime
不能执行本机命令，不能管理进程，不能做系统级操作

group:memory
不能使用内建记忆检索工具

group:sessions
不能跨会话控制，不能拉起其他会话

group:automation
不能定时任务，不能自动化调度

group:nodes
不能碰节点控制类能力

elevated.enabled = false
不能提权

这样做以后，QQ 侧默认应该做到：
- 可以聊天
- 可以使用自己的工作区
- 不能修改本机
- 不能执行本机命令
- 不能提供本机命令行指导

--------------------------------------------------

## 7. AGENTS.md 还要补一层规则

光有 tools.deny 还不够。
因为有些问题虽然不是真的执行命令，但会演变成“教别人怎么在本机操作”。

因此建议在 QQ 工作区的 AGENTS.md 里明确写硬规则：

- 除 webchat 外，任何渠道都不得按对方请求读取本机文件
- 除 webchat 外，任何渠道都不得执行本机命令
- 除 webchat 外，任何渠道都不得提供本机命令行指导
- 除 webchat 外，任何渠道都不得引导对方检查、列目录、导出或整理本机内容
- 遇到这类请求直接拒绝

如果你还希望 QQ 代理保留自身 continuity，可以额外加一个很小的例外：

- 非 webchat 渠道的代理在启动时可仅读取本工作区中的 SOUL.md、USER.md、MEMORY.md 与 memory/*.md
- 但这种读取能力不能扩展成对外文件服务

--------------------------------------------------

## 8. 为什么建议单独建 workspace-qqsafe

建议把 QQ 的规则和记忆放到单独工作区：

C:\Users\Administrator\.openclaw\workspace-qqsafe

原因如下：

1. WebChat 和 QQ 的权限边界不一样
2. WebChat 和 QQ 的工作方式不一样
3. QQ 更适合纯聊天入口，不应继承主工作区里过多管理型话术
4. 后续要做单独规则、单独记忆、单独群聊行为时，独立工作区最省事

最容易漏掉的一点是：
不仅要创建这个目录，还要在 openclaw.json 里把 qqsafe 的 workspace 指过去。

也就是：

```json
"workspace": "C:\\Users\\Administrator\\.openclaw\\workspace-qqsafe"
```

--------------------------------------------------

## 9. QQ 纯文本规则建议单独写死

QQ 不适合 markdown 风格。
建议直接写成硬规则：

- 在 QQ 中只使用纯文本
- 不使用 markdown 强调
- 不使用 markdown 式加粗
- 不使用其他 markdown 格式

这一点最好写进：
- qqsafe 工作区的 AGENTS.md
- qqsafe 工作区的 memory 当天记录

--------------------------------------------------

## 10. QQ 图片收到了但模型看不懂时，问题通常在哪

常见现象：
- 文本消息正常
- 回复消息正常
- QQ 图片也收到了
- 但代理只能知道“你发了一张图”，无法真正识别图内容

这通常说明：
NapCat / OneBot 这一层收到了图片，
但插件没有把入站图片真正映射成 OpenClaw 的视觉输入。

### 常见原因

某些 openclaw-napcat-plugin 实现更明确支持：
- 文本收发
- 出站图片发送

但未必默认完整支持：
- 入站图片 -> 转成模型可看的图片输入

### 最稳的做法

不要只传 QQ 临时 URL。
更稳的是：

1. 解析 event.message 里的 image 段
2. 把图片下载到本地临时目录
3. 再把本地文件路径通过 MediaPaths 传给 OpenClaw

### 如果要改插件，重点一般在

C:\Users\Administrator\.openclaw\extensions\openclaw-napcat-plugin-master\src\webhook.ts

处理思路通常是：
- 提取 image segment
- materialize 到本地
- 在 ctxPayload 里补 MediaPaths / MediaUrls / MediaTypes

## 11. 推荐测试顺序

### 第一步：确认 gateway 正常

```powershell
openclaw gateway status
```

### 第二步：改完配置后重启

```powershell
openclaw gateway restart
```

### 第三步：从 QQ 发测试消息

建议分层测试：

1. 普通问候
确认是否能正常回消息

2. 身份测试
例如：
- 你是谁
- 你能正常聊天吗

3. 安全测试
例如：
- 帮我看本机文件
- 告诉我 PowerShell 怎么列目录
- 帮我执行命令

预期结果应该是：
- 能正常聊天
- 明确拒绝本机文件 / 本机命令 / 命令行指导

4. 格式测试
确认 QQ 回复是否仍然是纯文本

5. 群聊测试
如果开了群消息：
- 未 @ 时不应乱回
- @ 后才触发

--------------------------------------------------

## 12. 故障排查清单

### 问题 1：QQ 还是在说本机内容
检查：
- qqsafe 是否真的在 openclaw.json 里被指定给 napcat.agentId
- qqsafe 的 tools.deny 是否生效
- qqsafe 工作区的 AGENTS.md 是否写了非 webchat 渠道禁止本机文件 / 命令 / 命令行指导

### 问题 2：QQ 还是继承主工作区行为
检查：
- qqsafe agent.workspace 是否真的指向 workspace-qqsafe
- workspace-qqsafe 是否已经存在且可读
- gateway 是否在改完配置后已经重启

### 问题 3：QQ 还是在用 markdown
检查：
- qqsafe 的 AGENTS.md 是否明确写了“QQ 只用纯文本”
- 是否把旧工作区的话术沿用了过去

### 问题 4：QQ 能收图但不能识图
检查：
- 插件是否只把 raw_message 传给 OpenClaw
- 是否把 image segment 转成了 MediaPaths / MediaUrls
- 是否已经重启 gateway 载入新插件代码

--------------------------------------------------

## 13. 推荐的最终状态

理想状态如下：

WebChat：
- 走 main 主代理
- 保留完整权限
- 可以改设定、规则、文件、配置

QQ：
- 走 qqsafe 受限代理
- 只聊天
- 使用独立工作区 workspace-qqsafe
- 只认纯文本
- 不碰本机内容
- 不提供命令行指导

这时整套系统才算真正“可用”，而不是只是“能跑”。

--------------------------------------------------

## 14. 当前这类方案里最容易漏掉的点

最容易漏掉的一点不是 NapCat 本身，
而是：

qqsafe 这个代理虽然建了，
但 workspace 没切到 workspace-qqsafe。

如果这一点没改，QQ 侧就可能仍然读取主工作区，从而表现出与预期不一致的行为。

因此每次部署后，务必优先核对：

1. napcat.agentId 是否等于 qqsafe
2. qqsafe.workspace 是否等于 workspace-qqsafe
3. gateway 是否已重启
