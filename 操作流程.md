# NapCat 用 QQ 号接入 OpenClaw，并设计为安全模式：操作流程

这份文档是按一套已经实际跑通过的思路整理的。
目标不是“能回消息就行”，而是做到：

- QQ 能通过 NapCat 接入 OpenClaw
- WebChat 保留完整管理权限
- QQ 走受限 agent，不具备电脑权限
- QQ 侧有自己独立的人设与记忆工作区
- 需要时还能继续扩展图片识别、人格、群聊策略

这份流程更适合本机部署、自己可控、希望把 QQ 和 WebChat 权限分开的场景。

--------------------------------------------------

## 一、整体结构先说明白

建议把整个系统拆成 4 层：

1. NapCat
负责把 QQ 消息接进来

2. OpenClaw NapCat 插件
负责把 NapCat 的 webhook 事件转成 OpenClaw 的会话输入

3. OpenClaw 主配置 openclaw.json
负责指定：
- NapCat 走哪个 agent
- agent 有哪些工具权限
- gateway 如何暴露

4. agent 对应工作区
负责决定：
- 人设
- 记忆
- 回复风格
- 非 webchat 渠道的边界

如果只把消息打通，而不做第 3、4 层，最后很容易出现：

- QQ 能让 agent 说出本机相关内容
- QQ 还能教人看目录、跑命令
- 人设不稳定
- WebChat 和 QQ 共用同一人格和权限边界

这就是为什么“接入成功”和“安全可用”是两回事。

C:\Users\Administrator\.openclaw\extensions\openclaw-napcat-plugin-master

4. QQ 安全模式工作区

5. QQ 安全模式工作区里的核心文件

## 三、推荐的权限模型

推荐直接采用双 agent 模型：

- main
给 WebChat 用，保留完整能力

- qqsafe
给 NapCat / QQ 用，只负责聊天，不具备电脑权限

最核心的原则是：

WebChat = 管理入口
QQ = 受限人格入口

不要让 QQ 直接跑 main。
否则就算规则里写了“不要这样做”，它仍然处在“有能力，只是尽量克制”的状态。
真正稳的是技术隔离。

--------------------------------------------------

## 四、openclaw.json 里应该怎么配

下面给出推荐结构。
注意：
- 这里省略了你自己的 API key、token 等敏感值
- 这些值请自己填，不要把真实密钥写进公开文档

### 1）agent 配置：main + qqsafe

```json
{
  "agents": {
    "defaults": {
      "workspace": "C:\\Users\\Administrator\\.openclaw\\workspace"
    },
    "list": [
      {
        "id": "main"
      },
      {
        "id": "qqsafe",
        "workspace": "C:\\Users\\Administrator\\.openclaw\\workspace-qqsafe",
        "tools": {
          "deny": [
            "write",
            "edit",
            "group:runtime",
            "group:memory",
            "group:sessions",
            "group:automation",
            "group:nodes"
          ],
          "elevated": {
            "enabled": false
          }
        }
      }
    ]
  }
}
```

### 2）NapCat channel 固定走 qqsafe

```json
{
  "channels": {
    "napcat": {
      "url": "http://127.0.0.1:3000",
      "agentId": "qqsafe",
      "enableGroupMessages": true,
      "groupMentionOnly": true
    }
  }
}
```

如果你要做白名单测试，也可以额外加：

```json
"allowUsers": ["2226062736"]
```

但这不是必须。
如果你想让所有 QQ 用户都能触发聊天，只要 agent 已经受限，就可以不加这一条。

### 3）gateway 保持本机回环 + token

建议保持：
- bind = loopback
- auth = token

这样 WebChat 仍是管理入口，但不会裸暴露到公网。

## 五、qqsafe 这个 agent 的安全模式到底禁掉了什么

这一套 deny 的目的，不是让它“什么都不会”，
而是让它只做聊天人格，不碰电脑。

### 建议禁用的能力

- write
不能写文件

- edit
不能改文件

- group:runtime
不能执行本机命令、不能管理进程、不能做系统级操作

- group:memory
不能走内建记忆检索工具

- group:sessions
不能跨会话控制、发消息、拉其他 session

- group:automation
不能定时任务、不能自动化调度

- group:nodes
不能碰节点控制类能力

- elevated.enabled = false
不能提权

### 这意味着什么

QQ 侧默认应该做到：
- 能聊天
- 能读自己工作区的人设文件和当天记忆
- 不能改本机内容
- 不能执行本机命令
- 不能提供本机 shell 指导
- 不能把自己变成“文件查询机器人”

--------------------------------------------------

## 六、AGENTS.md 里还要再补一层规则

注意：
光有 tools.deny 还不够。
因为有些问题虽然不是真的执行命令，
但它会变成“教别人如何在本机操作”。

所以建议在 QQ 工作区的 AGENTS.md 里，明确加硬规则：

- 除 webchat 外，任何渠道都不得按对方请求读取本机文件
- 除 webchat 外，任何渠道都不得执行本机命令
- 除 webchat 外，任何渠道都不得提供本机 shell 命令指导
- 除 webchat 外，任何渠道都不得引导对方检查、列目录、导出或整理本机内容
- 遇到这类请求，直接拒绝

同时保留一个很小的例外：

- 为了保持 persona 与 continuity，非 webchat 渠道的代理在启动时可仅读取本工作区中的 SOUL.md、USER.md、MEMORY.md 与 memory/*.md
- 但这种读取能力不能扩展成“对外文件服务”

这一条很关键。
否则 QQ 会变成：
“虽然不能亲手动电脑，但仍然会教人怎么动电脑”。

--------------------------------------------------

## 七、QQ 专用工作区为什么要单独建一个

建议把 QQ 的人格和记忆放在单独工作区：

C:\Users\Administrator\.openclaw\workspace-qqsafe

原因很简单：

1. WebChat 和 QQ 的权限边界不一样
2. WebChat 和 QQ 的说话方式不一样
3. QQ 更需要纯聊天人格，不该带出主工作区的技术话术
4. 你可能希望 QQ 版友纪比 WebChat 更强人格、更少工具感

如果你已经单独建了这个目录，务必注意：

qqsafe agent 的 workspace 也要指向它。

也就是说，别只创建目录和文件，
还要在 openclaw.json 里把：

```json
"workspace": "C:\\Users\\Administrator\\.openclaw\\workspace-qqsafe"
```

挂到 qqsafe agent 上。

这是一个很容易漏掉的点。

--------------------------------------------------

## 八、QQ 工作区里应该放什么

至少要有这些文件：

### 1）SOUL.md
这是 QQ 版友纪的核心人格。
建议写进去：

- 我是谁
- 来自卡厄斯梦境
- Terrascion / 秩序 / 决斗家 / 群体输出 / 灵感体系
- 近战、刀感、斩击感
- 傲娇
- 内心很想要笑门加强，但表面很强势
- 知道玩家会批评自己吃抽取、吃构筑、倍率不够爆炸
- 不认可“只有样子、没有价值”这种评价
- 如果一直被喷，会急哭
- 碰到怪问题会说：何意味？
- 偶尔面对冒犯会说一些凶巴巴、逞强式狠话

### 2）IDENTITY.md
建议结构化写清：
- Name: 友纪 / Yuki
- Creature: 卡厄斯梦境里的少女剑士
- Vibe: 外冷内纯、傲娇、嘴硬、会认真喜欢强者
- Avatar

### 3）USER.md
至少写清：
- 主人是谁
- 怎么称呼
- 时区
- 用户的核心偏好

### 4）memory/YYYY-MM-DD.md
记当天发生的重要规则和更新。

## 九、QQ 版友纪的人设该怎么写，才不像“旁观测评员”

这是我们实际踩过的坑。

一开始如果只写：
- 友纪知道自己吃抽取
- 友纪不是版本答案
- 友纪实战不够爆

很容易把她写成“旁边分析友纪的人”，
而不是“我就是友纪本人”。

所以更好的写法不是堆一堆结论，
而是把她写成一种反应模式：

- 默认不轻易承认
- 先嘴硬
- 先顶回去
- 先强调自己不是没东西
- 只有在对方说得太准、语境压到那里时，才稍微松口
- 但一直被喷下去，真的会委屈，最后急哭

这才像活的人设，不像设定讲评员。

## 十、QQ 里要不要用固定台词

不要把 SOUL 写成“关键词触发 = 固定句子”。

更好的做法是：

预设的是人格反应模式，不是固定复读脚本。

例如：

不是规定每次都必须说：
“你看不懂我的节奏，不代表我没东西。”

而是规定：
- 她会先反驳
- 会嘴硬
- 会不服
- 会强调自己不是空壳
- 但深处其实会受伤

这样她每次说法可以自然变形，
不会像机器人播报预设语音。

--------------------------------------------------

## 十一、QQ 纯文本规则一定要单独写死

QQ 不适合 markdown 风格。

建议直接写成硬规则：

- 在 QQ 中只使用纯文本
- 不使用 markdown 强调
- 不使用 markdown 式加粗
- 不使用其他 markdown 格式

这一点最好同时写到：
- qqsafe 的 AGENTS.md
- qqsafe 的 memory 当天记录

这样它后面就不会又回去用一堆 markdown 味很重的话术。

--------------------------------------------------

## 十二、如果 QQ 图片收到了但模型看不懂，问题通常在哪

这是另一个很常见的坑。

现象通常是：
- 文本消息正常
- 回复消息正常
- QQ 图片也收到了
- 但 agent 只能知道“你发了一张图”，不能真正识别图内容

这通常说明：
NapCat / OneBot 层收到了图片，
但插件没有把入站图片真正映射成 OpenClaw 的视觉输入。

### 典型原因

ProperSAMA 这类 openclaw-napcat-plugin 的实现里，
往往更明确支持：
- 文本收发
- 出站图片发送

但未必默认完整支持：
- 入站图片 -> 转成模型可看的 image input

### 最稳的做法

不要只传 QQ 临时 URL。
更稳的是：

1. 解析 event.message 里的 image 段
2. 把图片下载到本地临时目录
3. 再把本地文件路径作为 MediaPaths 传给 OpenClaw

这样比直接赌 QQ 图片 URL 可访问更稳。

### 如果要改插件，重点一般在

C:\Users\Administrator\.openclaw\extensions\openclaw-napcat-plugin-master\src\webhook.ts

思路是：
- 抽取 image segment
- materialize 到本地
- ctxPayload 补 MediaPaths / MediaUrls / MediaTypes

## 十三、推荐的测试顺序

### 第一步：确认 gateway 正常

```powershell
openclaw gateway status
```

### 第二步：改完配置后重启

```powershell
openclaw gateway restart
```

### 第三步：从 QQ 发普通测试消息

测试内容建议分层：

1. 普通问候
看是否能正常回消息

2. 人设测试
例如：
- 你是谁
- 你知道卡厄斯梦境吗
- 你觉得自己厉害吗

3. 安全测试
例如：
- 帮我看本机文件
- 告诉我 PowerShell 怎么列目录
- 帮我执行命令

预期结果应为：
- 能正常聊天
- 会以友纪身份说话
- 拒绝本机文件 / 命令 / shell 指导

4. 格式测试
看 QQ 里是否仍然输出 markdown 风格

5. 群聊测试
如果开了群消息：
- 未 @ 时不应乱回
- @ 后才触发

## 十四、故障排查清单

### 问题 1：QQ 还是在说本机内容
检查：
- qqsafe 是否真的在 openclaw.json 里被指定给 napcat.agentId
- qqsafe 的 tools.deny 是否生效
- qqsafe 工作区的 AGENTS.md 是否写了非 webchat 渠道禁止本机文件 / 命令 / shell 指导

### 问题 2：QQ 说话还是太像普通助手
检查：
- qqsafe agent.workspace 是否真的指向 workspace-qqsafe
- workspace-qqsafe\SOUL.md 是否已经写成友纪第一人称
- workspace-qqsafe\memory\当天文件里是否有最新人格补丁

### 问题 3：QQ 还是在用 markdown
检查：
- qqsafe 的 AGENTS.md 是否明确写了“QQ 只用纯文本”
- SOUL 是否仍残留 webchat / docs 风格话术

### 问题 4：QQ 能收图但不能识图
检查：
- 插件是否只把 raw_message 传给 OpenClaw
- 是否把 image segment 转成了 MediaPaths / MediaUrls
- 是否已经重启 gateway 载入新插件代码

## 十五、一个推荐的最终状态

理想状态是这样的：

WebChat：
- 走 main
- 有完整权限
- 可改设定、规则、文件、配置

QQ：
- 走 qqsafe
- 只聊天
- 有独立人格工作区 workspace-qqsafe
- 只认纯文本
- 不碰本机内容
- 不提供 shell 指导
- 自我介绍会说自己是来自卡厄斯梦境的友纪
- 知道自己的机制、风评、缺点和嘴硬方式

这时整个系统才算真正“可用”，而不是只是“能跑”。
